🔜 Backlog — Prioritized

High SSE streaming — user sees tokens as they arrive Latency · UX
Biggest single UX improvement available. Currently the user stares at a spinner for 2–7 seconds while all Groq rounds complete. With SSE, text and tool results stream in as they're generated.
Backend: text/event-stream response; push each chunk and tool result as an SSE event. Frontend: consume EventSource and append tokens progressively to the chat bubble.
Medium Parallel tool execution — multiple tools in one round Latency
When Groq returns multiple tool calls in one message, they are executed sequentially today. Parallelising with goroutines would cut latency proportionally to the number of concurrent tool calls.
Detect multiple tool_calls in a single Groq response → fan out to goroutines with a WaitGroup → collect results → feed back to Groq in a single message batch.
Medium Structured tool error responses — better LLM recovery Reliability
Tool errors are currently returned as plain strings. The model guesses how to recover. A structured { "error": "code", "message": "...", "suggestion": "..." } format would let the model give a more accurate user-facing explanation.
Define a ToolError struct in Go; return it as JSON in the tool result content. Update system prompt to describe the error shape.
Medium "Remove the gauge" — remove by widget type, not title UX · Tool
remove_widget requires exact title match. "Remove the gauge" fails because there is no widget titled "gauge." Users naturally refer to widgets by type, not by the configured title.
Extend remove_widget to accept a widget_type fallback. If title match fails, match by type within the dashboard — with a confirmation step if multiple widgets of that type exist.
Low Auto-update conversation title UX
All conversations are titled "New Conversation." Users with multiple open threads cannot tell them apart in the sidebar.
After the first user message, call Groq with a tiny prompt: "Summarise this chat in 5 words." Write the result to ai_conversations.title. No agentic loop needed — plain call.
Low get_machine_fields — "what can CW-01 measure?" Tool gap
"What metrics does CW-01 have?" today calls get_machines, which returns all machines. The model filters the result but sends unnecessary data. A dedicated tool would be leaner and more accurate.
Add get_machine_fields(machine_id) → returns only the field schema for one machine. Saves tokens and removes the need for the model to filter a full org-wide machine list.

🔭 Future Ideas

🔔
Alert management via AI
Allow the AI to acknowledge and resolve alerts. Currently blocked by design — needs a clear safety model before enabling write access to alert state.
🌍
Multi-machine comparison
"Compare speed of CW-01 and CW-02 last hour." Requires multi-machine tool calls and a combined response format.
📅
Scheduled reports
"Send me a daily summary of CW-01 production at 8am." Requires a cron/notification layer outside the current chat loop.
🤖
Anomaly detection
Let the AI proactively surface unusual patterns in telemetry — not just answer questions but flag issues before alerts fire.
🔗
MCP integration
Expose IotVision tools via the Model Context Protocol so external Claude instances or other AI clients can query factory data.
🎙️
Voice input
Factory floor operators often have gloved hands. Browser Web Speech API → transcript → existing chat endpoint. No backend change needed.
