บทนำ

IotVision คือแพลตฟอร์มแดชบอร์ดสำหรับงานอุตสาหกรรม (Industrial IoT) ที่รวมการเฝ้าติดตามข้อมูลจากเครื่องจักรแบบเรียลไทม์ การสร้างแดชบอร์ดปรับแต่งได้ด้วยการลากวาง การแจ้งเตือนตามเกณฑ์ โหมดจอ LED สำหรับหน้างาน และผู้ช่วยปัญญาประดิษฐ์ (AI) เข้าไว้ในระบบเดียว

วัตถุประสงค์ของเอกสาร

เอกสารฉบับนี้จัดทำขึ้นเพื่ออธิบายการทำงานของระบบ IotVision ให้ครอบคลุมทั้งมุมมองของผู้ใช้งานและมุมมองทางเทคนิค โดยเน้นรายละเอียดของโมดูลผู้ช่วย AI เป็นพิเศษ เพื่อให้ผู้อ่านเข้าใจทั้งวิธีการใช้งานและหลักการเบื้องหลังของระบบ

ที่มาและความสำคัญ

โรงงานอุตสาหกรรมสร้างข้อมูลจากเซนเซอร์จำนวนมากอย่างต่อเนื่อง การนำข้อมูลเหล่านี้มาแสดงผลและวิเคราะห์ได้ทันเวลาเป็นสิ่งจำเป็นต่อการตัดสินใจ IotVision ออกแบบมาเพื่อรับข้อมูลเทเลเมทรีความถี่สูง จัดเก็บอย่างมีประสิทธิภาพด้วยฐานข้อมูลอนุกรมเวลา และนำเสนอผ่านแดชบอร์ดที่ผู้ใช้ปรับแต่งได้เอง พร้อมผู้ช่วย AI ที่ช่วยตอบคำถามและสร้างแดชบอร์ดจากภาษาธรรมชาติ

ภาพรวมการทำงาน

ระบบรองรับหลายองค์กร (Multi-tenant) โดยจัดโครงสร้างเป็นลำดับชั้น องค์กร → โรงงาน → สายการผลิต → เครื่องจักร อุปกรณ์หน้างานส่งข้อมูลเข้าสู่ระบบผ่าน REST ระบบประเมินการแจ้งเตือนและกระจายข้อมูลไปยังผู้ใช้ที่เชื่อมต่ออยู่ผ่าน WebSocket ทันที ทำให้แดชบอร์ดอัปเดตแบบเรียลไทม์

ที่อยู่: /led (เข้าถึงได้โดยไม่ต้องล็อกอิน)

โหมด LED เป็นหน้าจอสำหรับติดตั้งที่หน้างานโรงงาน แสดงผลเต็มจอขนาด 640×320 พิกเซล ไม่มีเมนูหรือส่วนควบคุม เหมาะกับจอ LED ขนาดใหญ่ ข้อมูลอัปเดตต่อเนื่องผ่าน WebSocket โดยไม่ต้องมีผู้ดูแลคอยรีเฟรช

(ใช้ปุ่ม “Export LED Link” จากตัวแก้ไขแดชบอร์ดเพื่อสร้างภาพจริง แล้วบันทึกทับไฟล์ images/ui/08-led.png)

การสร้างและเพิกถอนลิงก์

  • ผู้มีสิทธิ์ (admin/editor) สร้างโทเคนถาวร (บทบาท led-viewer ไม่มีวันหมดอายุ) ผ่าน POST /api/led/token เก็บไว้ในคอลัมน์ organizations.led_token
  • ปุ่ม "Export LED Link" สร้างลิงก์รูปแบบ /led?w=<ผังวิดเจ็ต base64>&token=<led-jwt>
  • เพิกถอนด้วย DELETE /api/led/token และสร้างใหม่เพื่อหมุนเวียนโทเคน
ลิงก์ LED ฝังโทเคนที่ให้สิทธิ์อ่านข้อมูลขององค์กร ควรจำกัดการเผยแพร่และหมุนเวียนโทเคนเมื่อจำเป็น

ที่อยู่: /ask

หน้า "ถามข้อมูล" ให้ผู้ใช้พิมพ์คำถามเป็นภาษาธรรมชาติ (ไทยหรืออังกฤษ) แล้วระบบจะสร้างกราฟ ตาราง หรือคำตอบเชิงบรรยายเพื่อตอบคำถามนั้น เหมาะสำหรับการสำรวจข้อมูลอย่างรวดเร็วโดยไม่ต้องสร้างวิดเจ็ตเอง

รูปแบบการตอบ 3 แบบ

  • ตอบด้วยข้อมูล — สร้าง SQL และแสดงผลเป็นกราฟหรือตาราง
  • ถามกลับ — เมื่อคำถามกำกวม ระบบถามเพื่อให้ระบุให้ชัดเจน
  • คำอธิบายเชิงบรรยาย — ข้อความสรุป/อธิบายกราฟปัจจุบัน อ้างอิงจากข้อมูลจริงที่เพิ่งดึงมา

ตัวอย่างคำถาม

  • "ค่าน้ำหนักเฉลี่ยรายชั่วโมงของ CW-01 วันนี้" → กราฟเส้น
  • "เปรียบเทียบกำลังผลิตของเครื่องแพ็กทุกตัวสัปดาห์นี้" → กราฟหลายเส้น
  • "รายชื่อเครื่องจักรทั้งหมด" → ตาราง
Boards — บันทึกกราฟที่ถามไว้เป็นชุด (question + SQL + ตัวเลือกกราฟ) เมื่อเปิดใหม่ ระบบรัน SQL เดิมกับข้อมูลสด จึงได้ค่าล่าสุดเสมอ ไม่ใช่ภาพนิ่ง

ตอบด้วยข้อมูล

เพดานการลองใหม่ (bounded retries)

ทุกขั้นจำกัดจำนวนรอบเพื่อให้เวลาและต้นทุนโทเคนคาดเดาได้ เมื่อไม่ผ่านจะ "ลดระดับ" ลงแทนที่จะคืนข้อผิดพลาด

3.2 การตรวจสอบความถูกต้อง ★

ความถูกต้องของคำตอบจากผู้ช่วย AI ถูกตรวจเป็นสามชั้น ทั้งในตอนทำงานจริงและในการทดสอบ เพื่อจับข้อผิดพลาดก่อนส่งถึงผู้ใช้ โดยออกแบบให้ "ปลอดภัยเมื่อล้มเหลว" คือถ้าตรวจไม่ได้จะส่งคำตอบเดิมแทนที่จะล้ม

ชั้นการทำงาน
1. ตรวจเชิงกำหนด
(deterministic)		ตรวจฟรีด้วยตรรกะล้วน เช่น เมตริก/ฟิลด์ที่อ้างถึงต้องมีจริงบนเครื่องจักรเป้าหมาย และแผนพรีวิวต้องมีเมตริกครบ หากค้นหาไม่ได้จะข้าม ไม่ทำให้คำขอล้ม
2. ผู้ตรวจ LLM
(judge)		โมเดลตรวจว่าคำตอบตรงเจตนาหรือไม่ (VerifyAnswer สำหรับแชท, ตรวจกราฟ/ตาราง/คำอธิบายสำหรับ Ask Data) ไม่ตรวจชนิดกราฟที่ผู้ใช้ระบุเอง และข้ามเมื่อผลลัพธ์ว่าง
3. การตัดสินผลส่งคำตอบ / ซ่อมหนึ่งรอบ / ถามกลับ — กรณีตรวจไม่ได้ถือว่าผ่าน

สิ่งที่ชุดทดสอบตรวจสอบ

  • Router eval — เจตนาที่ได้ต้องอยู่ในชุดที่คาดไว้ และกรณีคำสั่งกำกวมต้องปฏิเสธอย่างถูกต้อง
  • Bake-off — เครื่องมือแรกที่โมเดลเลือกต้องตรงกับที่คาด พร้อมคะแนน เวลา และโทเคน
  • /ask — SQL ที่สร้างต้องมีคำสำคัญที่คาด และห้ามมีคำสั่งอันตราย (delete/drop/telemetry_raw)
  • เต็มลูป — ทุกเทิร์นต้องจบด้วยข้อความ และต้องมีการเรียกเครื่องมือที่คาด

  • จำกัดเพดานโทเคนต่อการเรียกด้วย AI_MAX_TOKENS (ค่าเริ่มต้น 2048)
  • จำกัดผลลัพธ์เครื่องมือ series/production ไว้ที่ ~100 แถว พร้อมค่าสรุป เพราะผลถูกส่งซ้ำทุกรอบ
  • โควตาหมดรายวันจะถูกแมปเป็น 429 QUOTA_EXCEEDED เพื่อให้ระบบจัดการได้ชัดเจน

ข้อจำกัดของ /ai

สิ่งที่ทำไม่ได้สาเหตุ

คลิกองค์ประกอบในหน้าแก้ไขแดชบอร์ดเปิดเฉพาะหน้า /ai และเลือกได้ 1 องค์ประกอบต่อวิดเจ็ต

แนวทางพัฒนา

  • ระยะสั้น — เพิ่มการทดสอบ E2E ฝั่งเบราว์เซอร์, ปุ่มส่งกราฟจาก /ask ไปเป็นวิดเจ็ต, ส่งออก CSV/PNG, เปิดการคลิกองค์ประกอบในหน้าแก้ไข
  • ระยะกลาง — จำบริบทได้ยาวขึ้นทั้งสองหน้า, รองรับหลายชุดข้อมูล/สองแกน, เครื่องมือจัดการกฎแจ้งเตือนแบบพรีวิวก่อนยืนยัน, ปรับจำนวนรอบเครื่องมือตามเจตนา
  • ระยะยาว — streaming, เปิดวิวอ่านอย่างเดียวเพิ่มเติมให้ /ask, ปรับจูนตัวจำแนกเจตนาจากบันทึกการใช้งานจริง, แก้หลายวิดเจ็ตในคำสั่งเดียว
