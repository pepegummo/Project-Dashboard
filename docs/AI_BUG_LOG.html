1 Bug Dashboard name lookup — exact match only tool_actions.go ▾
Symptom"add widget to CW-01 Overview" silently fails when the dashboard's actual name is "Machine Overview — CW-01".
Root causeresolveDashboardID used LOWER(name) = LOWER($2) (exact). Machine lookup already used substring — inconsistent.
FixChanged to LIKE '%'||LOWER($2)||'%' with exact-match ranked first via CASE.
WHERE organization_id = $1 AND LOWER(name) LIKE '%' || LOWER($2) || '%'
ORDER BY
  CASE WHEN LOWER(name) = LOWER($2) THEN 0 ELSE 1 END,
  created_at DESC
LIMIT 1
2 Bug preview_update_widget cannot change machine schema.go · dashboard_action.go ▾
Symptom"เปลี่ยน CW-01 เป็น CW-02 ใน widget นี้" — machine unchanged silently.
Root causeNo machine parameter in the tool schema or handler. Model had nowhere to pass the new machine name.
FixAdded Machine string `json:"machine"` to PreviewUpdateWidgetArgs. Handler resolves via resolveMachineID and writes changes["machine"] + changes["machineUuid"].

6 Optimization System prompt split — base + context extension controller.go ▾
ProblemEvery request included preview-editing instructions and the CONTEXT section even when no dashboard was open — ~100 wasted tokens per no-context request.
Root causeSingle systemPrompt constant always fully appended regardless of context.
FixSplit into systemPromptBase (always sent, cache-stable) and systemPromptContextExt (appended only when body.Context != "").

ProblemGreetings sent the full tool list (~870 tokens) to Groq even though no tool call would be needed.
Root causeNo mechanism to skip tools for non-actionable messages.

10 Discovery Omitting parameters field breaks Groq schema validation controller.go ▾
SymptomAfter slim schemas deployed, all show_metric calls errored: "additionalProperties 'machine', 'metric', 'viz' not allowed". Bake-off: 0/5 metric reads passing.
Root causeOmitting parameters from a tool definition makes Groq treat it as {"additionalProperties":false}, rejecting any args the model passes — even correct ones.
FixAdded "parameters": {"type":"object","additionalProperties":true} to toGroqToolSlim(). After fix: 13/13 bake-off cases pass.
func toGroqToolSlim(t map[string]any) map[string]any {
    return map[string]any{
        "type": "function",
        "function": map[string]any{
            "name":        name,
            "description": desc,
            "parameters": map[string]any{
                "type":                 "object",
                "additionalProperties": true,  // ← required
            },
        },
    }
}
11 Bug LineChart data unreadable by AI — missing get_telemetry_series tool schema.go · tool_actions.go · controller.go ▾
SymptomAI could not describe what a line chart was plotting. Asking "what does the speed trend show?" returned only a scalar summary (avg/min/max) from get_telemetry_trend — not the actual bucketed data points the chart renders.
Root causeNo AI tool wrapped telemetry.GetSeries(). The line chart widget calls /telemetry/{id}/series and receives [{bucket, avg, min, max, count}, …] — a completely different shape from /aggregate which returns one row.
FixAdded get_telemetry_series tool. Wraps tk.tel.GetSeries(ctx, id, metric, timeRange, …). Returns all data points the chart plots. Parameters: machine_id, metric, time_range (5m–30d).
// tool_actions.go
func (tk *ToolKit) GetTelemetrySeries(ctx, orgID, raw) {
    return tk.tel.GetSeries(ctx, id, args.Metric, timeRange, "", "", &orgID)
    // → [{bucket, avg, min, max, count}, …]
}
12 Bug DailyCount widget data mismatch — get_daily_count called wrong service method schema.go · tool_actions.go · controller.go ▾
Symptom"How many pieces did CW-01 produce in the last 48 half-hour buckets?" returned 7 per-calendar-day rows — not the 48 half-hour buckets the widget was displaying.
Root causeget_daily_count calls GetDailyCount() → /daily-count?days=7. The DailyCount widget uses GetBucketCount() → /count?bucket=30m&points=48&status=all — a separate endpoint with bucket size, point count, SKU, and status filters. The two never overlapped.
FixAdded get_production_count tool. Wraps tk.tel.GetBucketCount(ctx, id, sku, status, bucket, points, &orgID). Returns [{bucket, count}] matching exactly what the widget shows. Parameters: machine_id, bucket (e.g. "30m"), points (default 48), sku, status.
// tool_actions.go
func (tk *ToolKit) GetProductionCount(ctx, orgID, raw) {
    return tk.tel.GetBucketCount(ctx, id, args.Sku, args.Status,
        args.Bucket, args.Points, &orgID)
    // → [{bucket, count}, …] — same shape as widget
}
