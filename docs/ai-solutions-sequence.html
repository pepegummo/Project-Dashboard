answer_with_widget is the key new tool. It takes the AI's computed answer and a widget spec, and returns both together so the frontend can render text + widget in a single chat turn.

What the keyword hint was doing — and why it breaks

sequenceDiagram participant U as User participant AP as AI Handler (old) participant G as Groq Note over AP: wantsBuilderTools(message) — runs on every message U->>AP: "how many pieces in range 100-200?" AP->>AP: scan for keywords: "create","add","สร้าง","เพิ่ม",... Note over AP: ❌ "how many pieces" has no builder keyword
→ includeBuilder = false
→ only read tools sent to Groq U->>AP: "แสดง output ช่วงวันนี้" AP->>AP: scan for "แสดง"... not in keyword list Note over AP: ❌ misses — "แสดง" means "show" in Thai
→ builder tools excluded again U->>AP: "show output graph" AP->>AP: "show" not in keyword list (only "create","add",...) Note over AP: ❌ misses — user wants a widget but hint doesn't trigger Note over AP,G: New approach: graph tells AI what's available.
answer_with_widget is always in the tool list.
AI decides when to produce a widget — no keyword needed.
Root cause: keyword hints were gating builder tools (create/edit). But query-to-widget doesn't need builder tools at all — it needs answer_with_widget which is always available. The keyword gate was solving the wrong problem.
