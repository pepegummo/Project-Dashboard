🛠 Tool Catalogue

📡 Read — Telemetry
show_metric Render a live widget card (gauge / KPI / trend) for one metric. Handles count metrics → daily-count widget. Falls back to all available fields if metric name is ambiguous. Viewer
get_telemetry_trend Single aggregate (avg / min / max / count) over a time window — 5m to 30d. Used for text-based questions like "what was the average last hour?" Viewer
get_telemetry_series All time-bucketed data points a line chart renders. Returns [{bucket, avg, min, max, count}, …] — same shape as the widget. Used to read or describe what a chart is showing. Viewer
get_daily_count Per-calendar-day production count for one machine. Returns up to N days of {date, count} rows. Viewer
get_production_count Bucket-level piece counts with flexible bucket size (1m – 1d), point count, SKU filter, and status filter (all / good / reject). Mirrors exactly what the DailyCount widget displays. Viewer
get_active_alerts All open alert events for the org — machine, metric, value, severity, trigger time. Read-only; the AI cannot acknowledge or resolve alerts. Viewer
🏗 Read — Structure
get_machines Org machine list with names, types, statuses, and available metric field schema. Viewer
list_dashboards All dashboards in the org with names, widget counts, and URLs. Used before modify operations to resolve a dashboard by name. Viewer
✏️ Write — Dashboard Preview (no DB write)
preview_dashboard Generate a template widget set (machine_overview / machine_production / machine_maintenance) and render it as a preview card with Confirm / Discard buttons. No database write. Viewer
preview_add_widget Add a widget to the in-progress preview plan. Only available when a preview is open. Viewer
preview_update_widget Edit any field of a widget in the preview — title, machine, metric, unit, min/max, date range, bucket, SKU, status. Viewer
preview_remove_widget Remove a widget from the preview by its current title. Viewer
🔒 Write — Confirmed (DB write · Editor / Admin only)
create_custom_dashboard Save a confirmed preview as a real dashboard. Never exposed to the LLM — only the frontend calls this after the user clicks Confirm. Prevents accidental creation. Editor+
add_widget_to_dashboard Add one widget to an already-saved dashboard, matched by name (substring). Writes immediately — no preview step. Editor+
remove_widget Remove a widget from a saved dashboard by title (case-insensitive match). Editor+

🔐 Security Model — 3 Layers

1
Org scoping at query level
Every tool resolves machine IDs through requireMachineInOrg(). A user can never query another org's telemetry — the check runs inside the service, not just at the API boundary.
2
Role gate at dispatch
Write tools check user.Role in dispatch(). Even if the LLM hallucinates a write call from a viewer session, the backend rejects it before any DB operation runs.
3
LLM never writes to DB directly
create_custom_dashboard is not in AllTools() — the LLM cannot call it. Only the frontend POST after the user clicks Confirm triggers the write. The preview step is not skippable.

📝 System Prompt Structure

Base prompt — always sent cache-stable
  • Language rule: reply in user's language
  • Tool call rules (when to call vs. answer in text)
  • Machine / dashboard name resolution
  • Alert limitation (read-only)
  • Date range support for chart widgets
  • Multi-turn behavior: don't repeat context already given
  • 13 explicit rules numbered for reference

~27 lines · Groq caches this across all requests.

Context extension — when dashboard open conditional
  • Preview-edit rules (preview_add/update/remove)
  • Live value forcing (show_metric required for readings)
  • @Widget Title token handling
  • Dashboard name context injected inline

~15 extra lines · only sent when body.Context != "". Saves ~100 tokens on no-context requests.
